## 三、可共用参数整合分析

### 3.1 门店数据模型（核心共用）

三个接口返回同一门店的不同字段子集，应统一为单一模型：

```
┌─────────────────────────────────────────────────────┐
│              统一门店模型 KTVStore                     │
├────────────┬──────────┬──────────────────────────────┤
│ 字段        │ 类型      │ 说明                         │
├────────────┼──────────┼──────────────────────────────┤
│ id         │ string   │ 门店ID（统一string）           │
│ title      │ string   │ 门店名称                      │
│ district   │ string   │ 城区slug（qingxiu等）          │
│ address    │ string   │ 门店地址                      │
│ phone      │ string   │ 门店电话                      │
│ cover      │ string   │ 封面图URL（列表/推荐用）        │
│ images     │ string[] │ 详情图URL数组（详情页用）       │
│ package    │ string   │ 套餐信息                      │
│ description│ string   │ 门店描述                      │
│ time       │ string   │ 更新时间（ISO日期）            │
└────────────┴──────────┴──────────────────────────────┘
```

各接口返回字段对照：

| 字段 | featured(首页) | list(城区) | detail(详情) | recommend(推荐) |
|------|:-:|:-:|:-:|:-:|
| id | ✅ | ✅ | ✅ | ✅ |
| title | ⚠️用`name` | ✅ | ✅ | ✅ |
| district | ✅ | — | ✅ | — |
| address | ✅ | ✅ | ✅ | — |
| phone | ❌缺 | ✅ | ✅ | — |
| cover | ✅ | ❌缺 | — | ✅ |
| images | — | — | ✅ | — |
| package | ❌缺 | ✅ | ✅(未渲染) | — |
| description | ❌缺 | ✅ | ✅ | ✅ |
| time | ❌缺 | ✅ | ✅ | — |

**问题汇总**：

| # | 问题 | 影响 | 建议 |
|---|------|------|------|
| 1 | 首页用`name`，其他用`title` | 前端渲染逻辑不统一 | **统一为`title`** |
| 2 | `id`类型不一致：list用number，其他用string | 后端模型歧义 | **统一为string** |
| 3 | 城区list接口缺`cover` | 城区页无法展示门店封面图 | **补充cover字段** |
| 4 | 详情页Mock有`package`/`phone`但未渲染到页面 | 套餐和电话信息丢失 | **详情页应渲染这两个字段** |
| 5 | 推荐接口返回了time/address/package/phone但未渲染 | 传了多余数据 | 只返回`id/title/description/cover`即可 |

### 3.2 全站可共用参数

| 共用参数 | 类型 | 出现页面 | 当前问题 | 整合建议 |
|----------|------|----------|----------|----------|
| **联系电话 phone** | string | 3页×3处≈9处 | district/detail未调API | 所有页面引入`main.js`的`loadContact()` |
| **网站标题 title** | string | 3页 | 仅index调API | 所有页面调用`/api/site-title` |
| **城区映射 DISTRICT_MAP** | object | 3处独立定义 | main.js/detail.js缺desc/count | 合并为`/api/districts`或共享JS文件 |
| **微信二维码 wechatQR** | string | 3页占位 | 无API | 新增`/api/site-config`统一返回 |
| **城区背景图 districtCover** | string | index.html | 仅青秀区有值，硬编码 | 纳入城区映射的`cover`字段 |
| **escapeHtml()** | function | 3处重复定义 | 代码冗余 | 抽取到`js/common.js` |
| **fetchJSON()** | function | 仅main.js定义 | detail.js/district.html无法复用 | 抽取到`js/common.js` |

### 3.3 建议合并为统一配置接口

将 `site-title` + `contact` + `wechatQR` 合并为单次请求：

```
GET /api/site-config
```

```json
{
  "title": "南宁KTV — 南宁七区KTV信息大全",
  "phone": "0771-7654321",
  "wechatQR": "https://cdn.example.com/qr.png"
}
```

**好处**：3个独立请求→1个请求，减少网络开销，全站统一。

### 3.4 建议新增城区列表接口

```
GET /api/districts
```

```json
{
  "list": [
    { "slug": "qingxiu", "name": "青秀区", "desc": "南宁CBD核心区域", "count": 12, "cover": "https://..." }
  ]
}
```

**替代**：3处硬编码的 `DISTRICT_MAP` / `pinyinMap`，新增 `cover` 字段供首页城区卡片使用。

## 四、最终建议的API架构

```
GET /api/site-config          → 全站配置（title/phone/wechatQR）
GET /api/districts            → 城区列表（slug/name/desc/count/cover）
GET /api/ktv/featured?limit=8 → 热门门店（id/title/district/address/cover）
GET /api/ktv/list?district=xx → 城区门店（完整模型，含cover）
GET /api/ktv/detail?id=xx&district=xx → 门店详情（完整模型+images）
GET /api/ktv/recommend?district=xx&count=5&excludeId=xx → 推荐（id/title/description/cover）
GET /api/articles?type=news   → 资讯列表（title/date/summary/district/ktvId）
```

统一门店模型字段：`id(string) | title | district | address | phone | cover | images[] | package | description | time`
